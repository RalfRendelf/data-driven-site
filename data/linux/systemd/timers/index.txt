<h2>systemd Timers: Замена cron для планирования задач в Linux</h2>

<p>
<strong>systemd Timers</strong> — это современный механизм планирования задач в Linux, который заменяет традиционный <code>cron</code>.
В отличие от <code>cron</code>, который использует отдельный демон, systemd Timers интегрированы в систему инициализации и позволяют планировать задачи с большей точностью и гибкостью.
В этой статье рассмотрены основы работы с systemd Timers, их синтаксис, примеры использования и сравнение с <code>cron</code>.
</p>

---

<h3>Теоретические основы</h3>

<h4>Что такое systemd Timers?</h4>
<p>
<strong>systemd Timers</strong> — это механизм планирования задач, который позволяет запускать сервисы или команды по расписанию.
Он состоит из двух компонентов:
</p>
<ul>
  <li><strong>Timer Unit</strong> — определяет, <strong>когда</strong> должна быть выполнена задача.</li>
  <li><strong>Service Unit</strong> — определяет, <strong>что</strong> должно быть выполнено.</li>
</ul>

<h4>Компоненты systemd Timers</h4>
<p>
Для работы systemd Timers требуются два файла:
</p>
<ol>
  <li><strong>Timer Unit</strong> (файл с расширением <code>.timer</code>) — определяет расписание.</li>
  <li><strong>Service Unit</strong> (файл с расширением <code>.service</code>) — определяет задачу.</li>
</ol>
<p>
Timer Unit ссылается на Service Unit с помощью директивы <code>Unit=</code>.
</p>

---
<h3>Синтаксис Timer Unit</h3>

<h4>Структура Timer Unit</h4>
<p>
Timer Unit состоит из нескольких секций:
</p>
<ul>
  <li><code>[Unit]</code> — общие настройки unit-файла.</li>
  <li><code>[Timer]</code> — настройки расписания.</li>
  <li><code>[Install]</code> — настройки установки.</li>
</ul>

<h4>Секция [Timer]</h4>
<p>
Секция <code>[Timer]</code> определяет, когда должна быть выполнена задача.
</p>

<table>
  <tr>
    <th>Директива</th>
    <th>Описание</th>
    <th>Пример</th>
  </tr>
  <tr>
    <td><code>OnBootSec=</code></td>
    <td>Время после загрузки системы</td>
    <td><code>OnBootSec=5min</code></td>
  </tr>
  <tr>
    <td><code>OnUnitActiveSec=</code></td>
    <td>Время после активации unit-файла</td>
    <td><code>OnUnitActiveSec=1h</code></td>
  </tr>
  <tr>
    <td><code>OnCalendar=</code></td>
    <td>Расписание в формате календаря</td>
    <td><code>OnCalendar=*-*-* 02:00:00</code></td>
  </tr>
  <tr>
    <td><code>OnMonotonicTimer=</code></td>
    <td>Таймер на основе монотонных часов</td>
    <td><code>OnMonotonicTimer=1h</code></td>
  </tr>
  <tr>
    <td><code>AccuracySec=</code></td>
    <td>Точность таймера</td>
    <td><code>AccuracySec=1min</code></td>
  </tr>
  <tr>
    <td><code>Unit=</code></td>
    <td>Unit-файл, который должен быть активирован</td>
    <td><code>Unit=mybackup.service</code></td>
  </tr>
  <tr>
    <td><code>Persistent=</code></td>
    <td>Запускать пропущенные задачи при следующем запуске</td>
    <td><code>Persistent=true</code></td>
  </tr>
  <tr>
    <td><code>RandomizedDelaySec=</code></td>
    <td>Случайная задержка перед запуском</td>
    <td><code>RandomizedDelaySec=10min</code></td>
  </tr>
</table>

<h3>Управление Timer Units</h3>

<h4>Просмотр списка таймеров</h4>
<p>
Для просмотра списка всех таймеров используйте команду:
</p>
<pre><code>
list_timers.sh
</code></pre>
<p>
Пример вывода:
</p>
<pre><code>
list_timers_output.txt
</code></pre>

<h4>Запуск, остановка и перезапуск таймеров</h4>
<p>
Для управления таймерами используйте команды <code>systemctl</code>:
</p>
<pre><code>
manage_timer.sh
</code></pre>

<h4>Включение и отключение таймеров</h4>
<p>
Для включения или отключения таймеров при загрузке системы используйте команды:
</p>
<pre><code>
enable_disable_timer.sh
</code></pre>

<h4>Просмотр статуса таймера</h4>
<p>
Для просмотра статуса таймера используйте команду:
</p>
<pre><code>
timer_status.sh
</code></pre>

<h4>Просмотр логов таймера</h4>
<p>
Для просмотра логов таймера используйте команду <code>journalctl</code>:
</p>
<pre><code>
view_timer_logs.sh
</code></pre>

<h4>Миграция с cron на systemd Timers</h4>
<p>
Для миграции с <code>cron</code> на systemd Timers выполните следующие шаги:
</p>
<ol>
  <li>Создайте <strong>Service Unit</strong> для задачи.</li>
  <li>Создайте <strong>Timer Unit</strong> с расписанием.</li>
  <li>Включите Timer Unit.</li>
  <li>Отключите задачу в <code>cron</code>.</li>
</ol>

<h5>Пример миграции</h5>
<p>
Предположим, у вас есть задача в <code>cron</code>:
</p>
<pre><code>
cron_example.txt
</code></pre>
<p>
Создайте <strong>Service Unit</strong>:
</p>
<pre><code>
backup_service_unit.txt
</code></pre>
<p>
Создайте <strong>Timer Unit</strong>:
</p>
<pre><code>
backup_timer_unit.txt
</code></pre>
<p>
Включите Timer Unit:
</p>
<pre><code>
enable_timer.sh
</code></pre>
<p>
Отключите задачу в <code>cron</code>:
</p>
<pre><code>
disable_cron.sh
</code></pre>

<h3>Практические рекомендации</h3>

<ul>
  <li><strong>Используйте <code>OnCalendar=</code></strong> для гибкого планирования задач.</li>
  <li><strong>Используйте <code>Persistent=true</code></strong> для запуска пропущенных задач.</li>
  <li><strong>Используйте <code>RandomizedDelaySec=</code></strong> для распределения нагрузки.</li>
  <li><strong>Используйте <code>AccuracySec=</code></strong> для настройки точности таймера.</li>
  <li><strong>Проверяйте логи</strong> с помощью <code>journalctl</code>.</li>
  <li><strong>Используйте <code>systemctl list-timers</code></strong> для просмотра всех таймеров.</li>
  <li><strong>Мигрируйте с <code>cron</code></strong> на systemd Timers для новых задач.</li>
</ul>

<h3>Особенности и подводные камни</h3>

<h4>Проблемы с синтаксисом OnCalendar</h4>
<p>
Если таймер не срабатывает, проверьте:
</p>
<ul>
  <li>Синтаксис <code>OnCalendar=</code>.</li>
  <li>Правильность указания <code>Unit=</code>.</li>
</ul>
<pre><code>
check_timer_syntax.sh
</code></pre>

<h4>Проблемы с правами доступа</h4>
<p>
Если задача не выполняется, проверьте:
</p>
<ul>
  <li>Права доступа к файлам и директориям.</li>
  <li>Пользователя, от имени которого выполняется задача.</li>
</ul>
<pre><code>
check_permissions.sh
</code></pre>

<h4>Проблемы с зависимостями</h4>
<p>
Если таймер не срабатывает из-за зависимостей, проверьте:
</p>
<ul>
  <li>Правильность указания зависимостей в <code>After=</code>.</li>
  <li>Статус зависимых сервисов.</li>
</ul>
<pre><code>
check_dependencies.sh
</code></pre>

<h4>Проблемы с Persistent</h4>
<p>
Если пропущенные задачи не выполняются, проверьте:
</p>
<ul>
  <li>Наличие директивы <code>Persistent=true</code>.</li>
  <li>Логи таймера:</li>
</ul>
<pre><code>
check_persistent.sh
</code></pre>

<h3>Заключение</h3>

<p>
<strong>systemd Timers</strong> — это современный и мощный механизм планирования задач в Linux, который предоставляет больше возможностей по сравнению с традиционным <code>cron</code>.
Они интегрированы в systemd, поддерживают зависимости, ограничения ресурсов и предоставляют удобные средства для мониторинга и отладки.
</p>

    <td><strong>Service Unit</strong></td>
    <td>Определяет, что должно быть выполнено.</td>
  </tr>
  <tr>
    <td><strong>OnCalendar=</strong></td>
    <td>Расписание в формате календаря.</td>
  </tr>
  <tr>
    <td><strong>OnBootSec=</strong></td>
    <td>Время после загрузки системы.</td>
  </tr>
  <tr>
    <td><strong>OnUnitActiveSec=</strong></td>
    <td>Время после активации unit-файла.</td>
  </tr>

    <td><strong>Persistent=</strong></td>
    <td>Запускать пропущенные задачи.</td>
