<h2>Мультиплексирование ввода-вывода: epoll, select, poll</h2>

<p>
Мультиплексирование ввода-вывода (I/O Multiplexing) — это механизм, позволяющий одному процессу одновременно мониторить несколько файловых дескрипторов (сокетов, каналов, файлов) на предмет доступности для чтения или записи. Это критически важно для серверных приложений, обрабатывающих тысячи соединений без создания отдельного потока для каждого клиента.
</p>

<p>
В Linux существуют три основных механизма мультиплексирования: <strong>select()</strong>, <strong>poll()</strong> и <strong>epoll()</strong>. В этой статье рассмотрим их особенности, сравним производительность и покажем практические примеры использования.
</p>

---

<h3>Теоретические основы</h3>

<p>
Мультиплексирование решает проблему эффективной обработки большого количества соединений. Традиционный подход с созданием отдельного потока для каждого клиента имеет следующие недостатки:
</p>

<ul>
  <li>Высокое потребление памяти (каждый поток требует своего стека).</li>
  <li>Высокая нагрузка на планировщик ОС.</li>
  <li>Сложности с синхронизацией между потоками.</li>
</ul>

<p>
Мультиплексирование позволяет одному потоку обрабатывать множество соединений, переключаясь между ними по мере необходимости.
</p>

<h4>Ключевые концепции</h4>

<table>
  <tr>
    <th>Концепция</th>
    <th>Описание</th>
  </tr>
  <tr>
    <td>Файловый дескриптор</td>
    <td>Уникальный идентификатор открытого файла, сокета или канала.</td>
  </tr>
  <tr>
    <td>Событие</td>
    <td>Состояние дескриптора (доступность для чтения/записи, ошибка).</td>
  </tr>
  <tr>
    <td>Level-Triggered (LT)</td>
    <td>Уведомление приходит, пока состояние активно (например, есть данные для чтения).</td>
  </tr>
  <tr>
    <td>Edge-Triggered (ET)</td>
    <td>Уведомление приходит только при изменении состояния (например, переход из "нет данных" в "есть данные").</td>
  </tr>
</table>

<h3>Механизм select()</h3>

<p>
<strong>select()</strong> — классический механизм мультиплексирования, поддерживаемый во всех Unix-подобных системах. Он позволяет мониторить несколько файловых дескрипторов на предмет доступности для чтения, записи или ошибок.
</p>

<h4>Основные функции</h4>
<ul>
  <li><code>select()</code> — ожидание событий на множестве дескрипторов.</li>
  <li><code>FD_ZERO()</code>, <code>FD_SET()</code>, <code>FD_CLR()</code>, <code>FD_ISSET()</code> — работа с множеством дескрипторов.</li>
</ul>

<h4>Пример: TCP-сервер на select()</h4>
<pre><code>
select_tcp_server_c.txt
</code></pre>

<h4>Недостатки select()</h4>
<ul>
  <li>Ограничение на количество дескрипторов (<code>FD_SETSIZE</code>, обычно 1024).</li>
  <li>Низкая производительность при большом количестве дескрипторов (O(n) сложность).</li>
  <li>Необходимость пересоздавать множество дескрипторов после каждого вызова.</li>
</ul>

---
<h3>Механизм poll()</h3>

<p>
<strong>poll()</strong> — усовершенствованная версия <code>select()</code>, которая использует массив структур вместо битовых масок. Это позволяет поддерживать больше дескрипторов и упрощает код.
</p>

<h4>Основные функции</h4>
<ul>
  <li><code>poll()</code> — ожидание событий на массиве дескрипторов.</li>
</ul>

<h4>Структура pollfd</h4>
<pre><code>
pollfd_struct_c.txt
</code></pre>

<h4>Пример: TCP-сервер на poll()</h4>
<pre><code>
poll_tcp_server_c.txt
</code></pre>

<h4>Преимущества poll() над select()</h4>
<ul>
  <li>Поддержка большего количества дескрипторов (не ограничен <code>FD_SETSIZE</code>).</li>
  <li>Более эффективная проверка событий.</li>
  <li>Нет необходимости пересоздавать множество дескрипторов.</li>
</ul>

---
<h3>Механизм epoll()</h3>

<p>
<strong>epoll()</strong> — самый современный и эффективный механизм мультиплексирования в Linux (доступен с ядра 2.6). Он оптимизирован для работы с большим количеством дескрипторов и предоставляет O(1) сложность для добавления/удаления дескрипторов и проверки событий.
</p>

<h4>Основные функции</h4>
<ul>
  <li><code>epoll_create1()</code> — создание дескриптора epoll.</li>
  <li><code>epoll_ctl()</code> — управление дескрипторами в epoll (добавление, изменение, удаление).</li>
  <li><code>epoll_wait()</code> — ожидание событий.</li>
</ul>

<h4>Сравнение Level-Triggered и Edge-Triggered</h4>
<table>
  <tr>
    <th>Режим</th>
    <th>Описание</th>
    <th>Преимущества</th>
    <th>Недостатки</th>
    <th>Когда использовать</th>
  </tr>
  <tr>
    <td>Level-Triggered (LT)</td>
    <td>Уведомление приходит, пока состояние активно.</td>
    <td>Проще в использовании.</td>
    <td>Меньше производительность (может быть много уведомлений).</td>
    <td>Для большинства приложений.</td>
  </tr>
  <tr>
    <td>Edge-Triggered (ET)</td>
    <td>Уведомление приходит только при изменении состояния.</td>
    <td>Высокая производительность.</td>
    <td>Сложнее в использовании (нужно читать все данные сразу).</td>
    <td>Для высокопроизводительных приложений.</td>
  </tr>
</table>

<h3>Практические рекомендации</h3>

<ul>
  <li><strong>Используйте <code>epoll()</code> для высоконагруженных серверов</strong> (тысячи соединений).</li>
  <li><strong>Используйте неблокирующий режим</strong> для сокетов в Edge-Triggered режиме.</li>
  <li><strong>Читайте все доступные данные</strong> в цикле в Edge-Triggered режиме.</li>
  <li><strong>Устанавливайте таймауты</strong> для <code>epoll_wait()</code>.</li>
  <li><strong>Закрывайте дескрипторы корректно</strong> (удаляйте из epoll перед закрытием).</li>
  <li><strong>Используйте <code>EPOLLEXCLUSIVE</code></strong> в многопоточных приложениях.</li>
</ul>

---
<h3>Особенности и подводные камни</h3>

<h4>Проблемы с Edge-Triggered режимом</h4>
<p>
В Edge-Triggered режиме ядро уведомляет только один раз о доступности данных. Если вы не прочитали все данные, вы не получите повторное уведомление. Поэтому необходимо читать данные в цикле, пока <code>recv()</code> не вернёт <code>EAGAIN</code> или <code>EWOULDBLOCK</code>.
</p>

<h4>Проблемы с "Thundering Herd"</h4>
<p>
В многопоточных приложениях несколько потоков могут одновременно пробудиться для обработки одного события (например, нового соединения). Это приводит к лишним системным вызовам и конкуренции. Для решения используйте флаг <code>EPOLLEXCLUSIVE</code> (доступен в ядрах Linux 4.5+).
</p>

<h4>Утечка файловых дескрипторов</h4>
<p>
Всегда удаляйте дескрипторы из <code>epoll</code> перед закрытием:
</p>
<pre><code>
epoll_ctl_cleanup_c.txt
</code></pre>

<h3>Заключение</h3>

<p>
Мультиплексирование ввода-вывода — это ключевой механизм для создания высокопроизводительных сетевых приложений в Linux. <strong>epoll()</strong> является самым эффективным механизмом для работы с большим количеством соединений, но <strong>select()</strong> и <strong>poll()</strong> также имеют свои области применения.
</p>

<table>
  <tr>
    <th>Механизм</th>
    <th>Когда использовать</th>
  </tr>
  <tr>
    <td>select()</td>
    <td>Простые приложения с малым количеством дескрипторов (&lt;1024).</td>
  </tr>
  <tr>
    <td>poll()</td>
    <td>Средние приложения (1024–10,000 дескрипторов).</td>
  </tr>
  <tr>
    <td>epoll()</td>
    <td><strong>Высокопроизводительные приложения</strong> (тысячи соединений).</td>
  </tr>
</table>
